1. Executive Summary

1.1 What Is QueueSync?
QueueSync is a native macOS productivity utility that enables reliable, queued file and folder transfer from macOS to Linux desktops using SSH and rsync. It is designed to eliminate the friction of repetitive manual transfers while providing queue management, automatic retries, progress visibility, and persistent state.

1.2 Problem Statement
Current file transfer workflows on macOS require repeated manual scp or rsync commands with no queue management, no retry logic, and no visibility into transfer progress. The process breaks down at scale:
	•	No queue — each transfer must be manually triggered
	•	No retries — network failure means starting over
	•	No persistence — terminal closes and state is lost
	•	No progress — no visibility into large transfer status
	•	No history — no record of what was transferred

1.3 Solution
QueueSync wraps rsync over SSH in a queue-managed, retry-capable, persistent engine. The user adds files, starts the worker once, and QueueSync handles the rest — sequentially, reliably, with full logging.

1.4 Strategic Context (5 GB Storage Constraint)
Engineering Decision: CLI-First Architecture
Free Storage Available: ~5 GB
Full Xcode Requirement: 15 GB – 25 GB minimum
QueueSync Engine footprint: < 200 MB total
Decision: Build production-grade CLI engine first. Add SwiftUI GUI in Phase 2 when storage is available.
Outcome: Same core architecture, maximum ROI with current constraints.

1.5 Product Vision
AirDrop simplicity. SSH security. Queue reliability. Linux compatibility.

3. Software Requirements Specification (SRS)

3.1 Functional Requirements
FR-01: Queue Management
	•	The system shall allow adding individual files to the queue
	•	The system shall allow adding entire folders (recursive) to the queue
	•	The system shall persist the queue to disk after every mutation
	•	The system shall load the queue from disk on startup
	•	The system shall support reordering queue items
FR-02: Transfer Execution
	•	The system shall execute transfers using rsync over SSH
	•	The system shall execute one transfer at a time (sequential)
	•	The system shall emit progress events during transfer
	•	The system shall detect transfer success via rsync exit code
FR-03: Retry Logic
	•	On failure, the system shall retry up to 3 times
	•	Wait time between retries: 5 sec, 10 sec, 20 sec
	•	After 3 failures, item status shall be set to 'failed'
	•	Failed items shall remain in queue and be retryable manually
FR-04: Persistence
	•	queue.json — active queue items
	•	history.json — completed/failed transfer records
	•	config.json — SSH host, user, destination path
	•	logs.txt — timestamped event log
FR-05: Logging
	•	Every queue event shall produce a timestamped log entry
	•	Log format: YYYY-MM-DD HH:mm  EVENT  filename
	•	Logs shall never contain passwords or secrets

4. High-Level Design (HLD)

4.1 System Overview
QueueSync is a layered Swift CLI application. Each layer has a single responsibility and communicates through defined interfaces. This separation ensures the GUI layer (Phase 2) can be added without modifying core logic.

4.2 Layer Architecture
QueueSync — Layered Architecture
[ CLI Interface Layer ]  ← User commands: add, list, start, pause, logs
         ↓
[ Queue Manager ]        ← Add, remove, reorder, status management
         ↓
[ Transfer Manager ]     ← rsync execution, progress parsing, result handling
         ↓
[ Retry Engine ]         ← Failure detection, backoff scheduling, retry dispatch
         ↓
[ Storage Layer ]        ← queue.json, history.json, config.json read/write
         ↓
[ Logger ]               ← Timestamped event log (logs.txt)
         ↓
[ Config ]               ← SSH host, user, destination path

4.3 External System Boundary
QueueSync communicates with a single external system: a Linux host running OpenSSH Server. All communication uses the rsync-over-SSH protocol. No other network dependencies exist.
External Boundary
macOS Host  →  [ SSH / rsync ]  →  Linux Desktop
                                    OpenSSH Server
                                    Incoming Folder  (/home/user/Desktop/Incoming/)

5.2 Transfer Worker Loop
Worker Execution Flow
1.  Load queue from queue.json
2.  Filter items with status == .pending
3.  Take first item
4.  Set status = .transferring, save queue
5.  Execute rsync subprocess
6.  Parse stdout for progress (bytes, percentage)
7a. Exit code 0  →  status = .completed, log SUCCESS, remove from queue
7b. Exit code != 0  →  increment retries
       retries < 3  →  wait (5s / 10s / 20s), goto step 5
       retries == 3  →  status = .failed, log FAILED
8.  Take next pending item, repeat
9.  Queue empty → worker exits

5.3 rsync Command Template
rsync -avz --partial --progress --rsh='ssh -o StrictHostKeyChecking=no' \
  <source_path> <user>@<host>:<destination_path>

Flags used:
	•	-a  Archive mode (preserves permissions, timestamps, symlinks)
	•	-v  Verbose output (enables progress parsing)
	•	-z  Compression in transit
	•	--partial  Keep partially transferred files on failure (enables resume)
	•	--progress  Emit byte progress to stdout

6. Data Models & Schema

6.1 QueueItem Schema (queue.json)
{
  "id":        "uuid-v4-string",
  "path":      "/Users/mac/Videos/movie.mp4",
  "type":      "file" | "folder",
  "status":    "pending" | "transferring" | "completed" | "failed" | "paused",
  "progress":  0-100,
  "retries":   0-3,
  "createdAt": "2026-04-24T12:00:00Z",
  "startedAt": "2026-04-24T12:01:00Z",   // optional
  "finishedAt":"2026-04-24T12:03:00Z"    // optional
}

6.2 Config Schema (config.json)
{
  "host":        "192.168.1.10",
  "user":        "linuxuser",
  "destination": "/home/linuxuser/Desktop/Incoming/",
  "sshKeyPath":  "~/.ssh/id_rsa",
  "maxRetries":  3
}

6.3 History Item Schema (history.json)
{
  "id":         "uuid-v4-string",
  "path":       "/Users/mac/Videos/movie.mp4",
  "status":     "completed" | "failed",
  "startedAt":  "2026-04-24T12:01:00Z",
  "finishedAt": "2026-04-24T12:03:00Z",
  "retries":    0,
  "sizeBytes":  104857600
}

6.4 Log Format (logs.txt)
2026-04-24 12:00  ADD        movie.mp4
2026-04-24 12:01  START      movie.mp4
2026-04-24 12:01  PROGRESS   movie.mp4  45%
2026-04-24 12:03  SUCCESS    movie.mp4
2026-04-24 12:03  START      folder/
2026-04-24 12:04  RETRY(1)   folder/
2026-04-24 12:05  FAILED     folder/
6.5 Status State Machine
QueueItem Status Transitions
pending  →  transferring   (worker picks item)
transferring  →  completed  (rsync exit 0)
transferring  →  retrying   (rsync exit != 0, retries < 3)
retrying  →  transferring   (after backoff wait)
retrying  →  failed         (retries == 3)
transferring  →  paused     (user issues pause command)
paused  →  transferring     (user issues resume command)

11. Security Design

11.1 Transport Security
	•	All transfers encrypted via SSH (AES-256 in transit)
	•	rsync tunneled through SSH — no plain-text data on network
	•	SSH host key verification enforced by default
11.2 Authentication
	•	SSH key-based authentication required
	•	No passwords stored in config.json or logs
	•	Private key path stored in config — key itself never read by app
11.3 Log Security
	•	Logs contain only: timestamps, event names, file paths
	•	Logs never contain: passwords, SSH keys, or secrets
	•	Log files stored in local Data/ directory (not synced)
11.4 File Permission Model
	•	queue.json, config.json set to chmod 600 on creation
	•	Only the running user can read/write data files

13. Quick Start Guide

Step 1 — Initialize the project
mkdir QueueSync && cd QueueSync
swift package init --type executable

Step 2 — Set destination config
queuesync config set host 192.168.1.10
queuesync config set user linuxuser
queuesync config set dest /home/linuxuser/Desktop/Incoming/

Step 3 — Ensure SSH key is authorized on Linux
ssh-copy-id linuxuser@192.168.1.10

Step 4 — Add files and start
queuesync add ~/Downloads/movie.mp4
queuesync add ~/Projects/backup/
queuesync list
queuesync start

Step 5 — Monitor
queuesync logs
queuesync status

Step 6 — Build the binary
swift build -c release
cp .build/release/QueueSync /usr/local/bin/queuesync
